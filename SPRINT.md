# Sprint 16 — Milestone 11: Agent-addressable review — anti-AI slice

This Sprint migrates the **anti-AI family** onto the shared review contract M10 landed, preserving its one distinctive power — category-level bulk: `anti_ai_report` emits structured review items (a `review-id` anchor plus blank `- Decision:` / `- Decision-note:` fields per flagged instance) while keeping the per-scene `BULK eligibility:` declaration and per-category `BULK:` header grammar; `anti_ai_fix` consumes explicit decisions and legal bulk defaults through the shared validator; the `anti_ai` family flips to `adopted` in `agents/review-grammars.yaml` with its container shape settled so a pre-M11 positional report can never validate as proceed; and the `amanuensis-review` companion gains category queues and legal bulk capture. This is the second of the four one-milestone-per-sprint slices (M10–M13 in ROADMAP.md; the shared design lives in the completed M10 section there). Big-bang, as with compliance: after this Sprint the old positional anti-AI annotation format is an invalid input to `anti_ai_fix` and no compatibility path is kept.

## Background — what is and isn't wrong today

Established by inspection during planning; tasks should not re-derive this.

- **Anti-AI annotation is positional today.** Per-entry tokens are written after the quote, on the same line or the next (`agents/steps/anti-ai-report.md:279-284`); a per-category `BULK: <action>[: <instruction>]` header goes at the head of the category subsection (`agents/steps/anti-ai-report.md:286-301`), gated by the per-scene `BULK eligibility:` block the report emits (`agents/steps/anti-ai-report.md:49-65`). The fix step resolves each entry's effective annotation positionally (`agents/steps/anti-ai-fix.md:47-53`) and **tolerates an illegal bulk header** by treating it as if absent (`agents/steps/anti-ai-fix.md:53`) — under the contract that becomes invalid input instead. Its `review_pending` gate is whole-file (blocks only when *no* annotations and *no* bulk headers exist — `agents/steps/anti-ai-fix.md:34`, `:193`); the contract's per-unit gate is stronger, per the compliance model (`agents/steps/compliance-fix.md:141`).
- **The contract already defines the target grammar.** `agents/review-grammars.yaml:143-174` holds the full `anti_ai` block — tokens, payload rules, blank-means with the bulk-inheritance exception (`:166`), the two-layer bulk rules (`:170`), mixed-granularity counting (`:171`), and proceed-state (`:173`) — marked `adoption: pending` / `migrates_in: M11`. The fixture `examples/review/anti-ai.md` shows the exact target shape (book-form ids, `BULK: FIX: rewrite` with a per-entry `SKIP` override, `BULK: SKIP`, a not-permitted category decided per entry) and documents its post-flip ledger.
- **The validator's bulk logic already works — verified at planning.** Against a scratch grammar copy with the marker flipped: the fixture reproduces its documented ledger exactly (no findings; total 5, pending 1, decided 1, inherited-by-bulk 2, skipped 1; verdict pending-remain, exit 4), and a `BULK:` header on a category not declared `BULK permitted` yields an invalid finding while the blank item under it stays **pending**, never inherited (exit 3). No bulk or counting work is needed in `scripts/validate-review-artifact.sh`.
- **Gap found at planning: no container check for anti_ai.** A pre-M11 positional `anti-ai.md` (entries with no anchors) parses as zero units and validates as **proceed, exit 0** under a flipped marker — verified with a specimen. That is the silent-pass the compliance `container_pattern` keys exist to prevent (M10 ROADMAP note), and the grammar file's container comment (`agents/review-grammars.yaml:38-47`) explicitly defers settling the pending families' container shapes to their milestones. Anti-AI's category headings (`### Em Dashes` … `### Flagged Words`) share no prefix beyond `### `, and the legitimately unit-free heading `### Summary — Scene <id>` ends in a scene id, so the existing `container_exempt_suffix` mechanism cannot exempt it — a generic `container_exempt_prefix` key is needed. The full design (`container_pattern: "### "` + `container_exempt_prefix: "### Summary"` + a ~3-line script extension) was prototyped on scratch copies at planning: structured fixture ledger unchanged, old-format specimen flagged `unanchored violation block` (exit 3), compliance fixture regression clean. Appended `#### Applied:` / `#### Escalated:` / `#### Override applied:` blocks do not match the `### ` prefix and open no containers.
- **One category emits two top-level lines per instance.** Synonym Cycling's format is `- Passage: …` plus a sibling `- Cycled terms: …` (`agents/steps/anti-ai-report.md:177-181`). The parser attributes fields to a unit by its single anchored item line (`item_line_pattern: "^- "`, anchor-adjacency rule), so the retargeted format must carry exactly one top-level `- ` line per unit — `Cycled terms` nests under the `Passage` line. Every other category already emits one line per instance.
- **The report's per-scene Summary already counts review units.** Unlike compliance (where `CLEAN` blocks are not units and a separate `Review units emitted:` line was added), every anti-AI flagged instance is a review unit, so the existing `Total flags: N` line (`agents/steps/anti-ai-report.md:255-271`) *is* the unit count; no new summary line is needed.
- **Smoke recipes 6–8 hand-author old-format `anti-ai.md` fixtures** (`examples/smoke/README.md:237-395`) and become invalid specimens once anti-AI migrates. Recipes 9–10 (compliance, structured) are the model for validator-verdict expectations. Compliance shipped in M10 is the model for both step retargets: `compliance_report`'s append-path ordinal continuation (`agents/steps/compliance-report.md:41`) and `compliance_fix`'s validator invocation and per-unit gates (`agents/steps/compliance-fix.md:41-47`, `:141-142`).
- **The step set is unchanged.** No step is added or removed and no `step_id` changes, so `scripts/check-pipeline-state.sh`, both CI workflow ymls, and both `pipeline-state.md` files stay byte-for-byte unchanged.

## Conventions adopted by this Sprint

Locked at planning (starred items are owner decisions from this Sprint's planning session); tasks don't rediscover them.

- **★ The bare-`FIX` fallback stays fixer-level.** Categories with no bare-FIX rule (Negative Parallelism, phrase-level Significance Inflation, Synonym Cycling, Cadence tics, Animacy Projection — `agents/steps/anti-ai-fix.md:83-90`) keep treating a bare `FIX` as `ESCALATE`, recorded in the apply log. The validator stays category-agnostic: family-level token rules only, so it accepts bare `FIX` anywhere in the family. Rationale: the fallback is visible (an `Escalated:` block), never silent, and per-category payload keys would add real machinery to the M10-locked script and machine-read schema for a case the apply log already surfaces. Rejected alternative: per-category payload-required rules in `review-grammars.yaml` and the validator.
- **★ The companion writes legal `BULK:` headers.** A stated bulk decision is category-level decision capture — the skill's exact job. The skill writes the header at the head of the category subsection, only where the report's `BULK eligibility:` block declares the category permitted, then re-validates. Its write-surface hard rule expands accordingly. Rejected alternative: bulk headers stay human hand-edits (would leave the one decision type M11 adds outside the capture layer).
- **★ The progress ledger is unchanged.** The `inherited-by-bulk` bucket versus `pending`/`decided` already distinguishes item units from bulk-covered items, and the companion's category-summary view covers presentation; the M10 fixtures' documented expected ledgers stay valid untouched. Rejected alternative: an explicit `bulk-adjudicated categories: N` ledger line (touches the locked script and every fixture expectation for an informational count).
- **Container settlement: `anti_ai` gets `container_pattern: "### "` with a new generic `container_exempt_prefix: "### Summary"` key.** Without a container check, a pre-M11 positional report validates as proceed/exit 0 (verified at planning) — the silent-pass the compliance container check exists to prevent, and the invariant M10 shipped ("an old-format report can never pass as zero-units/proceed") extends to every migrated family. The exempt heading `### Summary — Scene <id>` ends in a scene id, so the existing suffix mechanism cannot match it; a prefix exemption is the minimal generic extension (~3 lines, prototyped at planning: fixture ledger unchanged, old format invalid, compliance regression clean). This is the one deliberate change to the M10-locked validator this Sprint; the ledger lock above is unaffected.
- **One top-level `- ` line per review unit.** The anchor sits immediately above the unit's single item line; auxiliary report lines and the `Decision:` / `Decision-note:` fields nest below it (one line = one fact, so the script's line-oriented parsing and anchor-adjacency rule hold). Synonym Cycling is the only category this changes: `- Cycled terms:` becomes a nested field under the `- Passage:` line.
- **The report emits blank decision fields everywhere, including bulk-permitted categories.** The report cannot know the human's bulk choice; the `BULK eligibility:` line's "recommended default" is the producer's recommendation, never a decision. A `BULK:` header is written only by the human (or the companion capturing the human's statement) during review.
- **An illegal bulk header now blocks; treat-as-absent is removed.** Today `anti_ai_fix` treats a `BULK:` header on a not-permitted category as if absent (`agents/steps/anti-ai-fix.md:53`). Under the contract it is `invalid` (validator exit 3) and blocks the step, per the ROADMAP M11 done-when ("invalid and never satisfies review evidence") and the compliance model for invalid input (`agents/steps/compliance-fix.md:142`). Rationale: a silently ignored header is a positional-convention smell — the human believes the category is adjudicated when it is not; `invalid` is a legal, reported state with a line-naming finding.
- **The per-unit pending gate replaces the whole-file gate,** exactly as compliance's did: any flagged instance with a blank `Decision:` not covered by a legal bulk header is a pending unit and blocks `anti_ai_fix` as `review_pending`, naming the pending review-ids. Deliberately stronger than today's all-bare-only gate (`agents/steps/anti-ai-fix.md:193`), for the same reason recorded in Sprint 15: silently skipped units are the accepted-vs-unreviewed ambiguity the milestone kills.
- **Anti-AI tokens and bulk grammar are unchanged; their *carrier* changes.** `FIX` / `FIX: <instruction>` / `SKIP` / `ESCALATE` move into the `Decision:` field; `BULK: FIX[: <instruction>]` / `BULK: SKIP` headers and the `BULK eligibility:` declaration survive verbatim. Big-bang: after this Sprint the positional form is an invalid input to `anti_ai_fix`.
- **No new Summary line.** Every anti-AI flagged instance is a review unit, so the per-scene `Total flags: N` already equals the units emitted; adding a `Review units emitted:` line (compliance's solution to CLEAN blocks not being units) would be redundant here.
- **Artifact-state mechanics are untouched.** Freshness stays a derived predicate; override stays stale-axis-only, anchor-gated, recorded in the apply log with the existing `Override:` block grammar (`agents/steps/anti-ai-fix.md:159-185`); `agents/review-validation.md` and `agents/orchestrator.md` already cover the structured evidence and bulk automation rules and are not edited this Sprint.
